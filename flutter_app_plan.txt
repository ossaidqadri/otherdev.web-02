# Flutter App Architecture Plan: otherdev Native Port

## Executive Summary

Complete native port of otherdev portfolio/agency website to Flutter (iOS, Android, macOS, Windows, Linux). Backend REST + SSE APIs stay on Next.js. Flutter app consumes `/api/chat/native` (new plain SSE endpoint), `/api/contact`, blog/work metadata via standard REST. No tRPC; backend RAG stays server-side; Flutter is a new client, not a replacement for the web.

**Key Principles:**
- Feature-first folder structure (not layer-first)
- Explicit types everywhere; no `dynamic`
- Riverpod 3 + code-gen for state
- Native UI with `shadcn_flutter` + Lucide icons
- Desktop as first-class citizen (not afterthought)
- Biome-equivalent Dart discipline

## 5. Backend API Changes (Next.js `/api/chat/native`)

### New Endpoint: `/api/chat/native`

**Purpose:** Plain W3C SSE (Server-Sent Events) for Flutter clients without Vercel's custom data-stream protocol.

**Route:** `web/src/app/api/chat/native/route.ts`

**Implementation Notes:**
- Accept same request body as `/api/chat/stream`
- Return `text/event-stream` with standard SSE format
- Each AI message chunk: `data: {"type":"text","content":"..."}\n\n`
- Tool calls: `data: {"type":"tool","name":"...","args":{...}}\n\n`
- Stream end: `data: [DONE]\n\n`

**Pseudocode:**
```typescript
export async function POST(request: Request): Promise<Response> {
  // 1. Rate limit (same as /api/chat/stream)
  // 2. Validate messages
  // 3. Open SSE writer
  // 4. Stream to ReadableStream (W3C EventSource compatible)

  const encoder = new TextEncoder()
  const stream = new ReadableStream({
    async start(controller) {
      try {
        // Use existing handleStreamChat but extract chunks
        const stream = await streamChatResponse(/* ... */)
        for await (const chunk of stream) {
          // Write SSE-formatted: data: {...}\n\n
          controller.enqueue(encoder.encode(`data: ${JSON.stringify(chunk)}\n\n`))
        }
        controller.enqueue(encoder.encode('data: [DONE]\n\n'))
        controller.close()
      } catch (error) {
        controller.error(error)
      }
    }
  })

  return new Response(stream, {
    status: 200,
    headers: {
      'Content-Type': 'text/event-stream',
      'Cache-Control': 'no-cache',
      'Connection': 'keep-alive',
      'X-RateLimit-Limit': ...,
      'X-RateLimit-Remaining': ...,
    }
  })
}
```

## 8. Authentication Flow (FlutterFire)

### Setup

**Firebase Console:**
1. Create iOS, Android apps in Firebase Console
2. Download `GoogleService-Info.plist` (iOS) тЖТ `ios/Runner/`
3. Download `google-services.json` (Android) тЖТ `android/app/`
4. Enable Authentication: Email/Password, Google Sign-In, Apple Sign-In

**lib/shared/data/auth_service.dart**

```dart
import 'package:firebase_auth/firebase_auth.dart';
import 'package:google_sign_in/google_sign_in.dart';

class AuthService {
  final _firebaseAuth = FirebaseAuth.instance;
  final _googleSignIn = GoogleSignIn();

  Stream<User?> get authStateChanges => _firebaseAuth.authStateChanges();

  User? get currentUser => _firebaseAuth.currentUser;

  Future<UserCredential> signUpWithEmail(String email, String password) async {
    try {
      return await _firebaseAuth.createUserWithEmailAndPassword(
        email: email,
        password: password,
      );
    } on FirebaseAuthException catch (e) {
      throw AuthFailure.fromException(e);
    }
  }

  Future<UserCredential> signInWithEmail(String email, String password) async {
    try {
      return await _firebaseAuth.signInWithEmailAndPassword(
        email: email,
        password: password,
      );
    } on FirebaseAuthException catch (e) {
      throw AuthFailure.fromException(e);
    }
  }

  Future<UserCredential> signInWithGoogle() async {
    try {
      final account = await _googleSignIn.signIn();
      if (account == null) throw AuthFailure.cancelled();

      final auth = await account.authentication;
      final credential = GoogleAuthProvider.credential(
        accessToken: auth.accessToken,
        idToken: auth.idToken,
      );

      return await _firebaseAuth.signInWithCredential(credential);
    } catch (e) {
      throw AuthFailure.fromException(e);
    }
  }

  Future<void> signOut() async {
    await Future.wait([
      _firebaseAuth.signOut(),
      _googleSignIn.signOut(),
    ]);
  }

  Future<void> resetPassword(String email) async {
    try {
      await _firebaseAuth.sendPasswordResetEmail(email: email);
    } on FirebaseAuthException catch (e) {
      throw AuthFailure.fromException(e);
    }
  }
}
```

// lib/shared/domain/entities/user.dart
@freezed
class User with _$User {
  const factory User({
    required String id,
    required String email,
    String? displayName,
    String? photoUrl,
  }) = _User;

// lib/shared/domain/failures/auth_failure.dart
sealed class AuthFailure {
  const AuthFailure();

  factory AuthFailure.fromException(dynamic e) {
    if (e is FirebaseAuthException) {
      return switch (e.code) {
        'email-already-in-use' => EmailAlreadyInUse(),
        'invalid-email' => InvalidEmail(),
        'weak-password' => WeakPassword(),
        'user-not-found' => UserNotFound(),
        'wrong-password' => WrongPassword(),
        _ => GenericAuthFailure(e.message ?? 'Unknown error'),
      };
    }
    return GenericAuthFailure(e.toString());
  }

  const factory AuthFailure.cancelled() = Cancelled;
}

final class EmailAlreadyInUse extends AuthFailure {}
final class InvalidEmail extends AuthFailure {}
final class WeakPassword extends AuthFailure {}
final class UserNotFound extends AuthFailure {}
final class WrongPassword extends AuthFailure {}
final class GenericAuthFailure extends AuthFailure {
  final String message;
  const GenericAuthFailure(this.message);
}
final class Cancelled extends AuthFailure {}

// lib/providers.dart
@riverpod
AuthService authService(AuthServiceRef ref) => AuthService();

@riverpod
Stream<User?> currentUser(CurrentUserRef ref) {
  final authService = ref.watch(authServiceProvider);

    return User(
      id: firebaseUser.uid,
      email: firebaseUser.email ?? '',
      displayName: firebaseUser.displayName,
      photoUrl: firebaseUser.photoURL,
    );
  });
}

@riverpod
String? authToken(AuthTokenRef ref) {
  final user = ref.watch(currentUserProvider);
  // In real app: get ID token from Firebase

  );

## 12. Key Design Decisions & Tradeoffs

### 1. No tRPC, Plain REST + SSE
**Decision:** Flutter calls plain REST endpoints + SSE streams.
**Rationale:**
- flutter_ai_toolkit is designed for OpenAI-compatible APIs + streaming
- tRPC has minimal Flutter support; REST is universal
- SSE is W3C standard; works natively with flutter_ai_toolkit

**Tradeoff:** Manual HTTP serialization vs. type-safe tRPC. Acceptable because API surface is small.

### 2. RAG Stays Server-Side
**Decision:** Qdrant/Cohere stay on Next.js backend.
**Rationale:**
- Flutter app doesn't need direct vector DB access
- Backend can update RAG logic without app release
- Reduces app permissions & bundle size

**Tradeoff:** Can't do offline RAG on device. Acceptable; chat requires internet anyway.

### 3. Riverpod 3 + Code-Gen (Explicit Override of Built-in State Management)
**Decision:** Riverpod 3 (`^3.0.0`) with `@riverpod` annotations and `Notifier` API. This is an explicit override of `rules.md`'s default (prefer built-in `ValueNotifier`/`ChangeNotifier`).
**Riverpod 3 key changes from v2:**
- `Ref` type is now generic тАФ provider-specific `Ref` subclasses (e.g. `CounterRef`) are removed; use plain `Ref` everywhere
- `StateProvider`, `StateNotifierProvider`, `ChangeNotifierProvider` moved to `/legacy.dart` тАФ do not use them; `Notifier`/`AsyncNotifier` is the only canonical API
- Code-gen annotation (`@riverpod`) and `_$ClassName` pattern are unchanged
**Why built-in solutions fall short for this app:**
- `AsyncValue<T>` natively wraps loading/error/data in one type тАФ you'd need 3 `ValueNotifier`s or a custom class to replicate this for every async operation
- `StreamProvider` turns the SSE AI chat stream into reactive state automatically; `StreamBuilder` requires manual `connectionState`, error, and reconnection handling per widget
- Provider composition: `chatProvider` depends on `authProvider` depends on `apiClientProvider` тАФ Riverpod wires these at declaration; `ChangeNotifier` requires manual constructor injection threading
- Providers have no `BuildContext` dependency тАФ they work in isolates, background services, and tests without a widget tree. Built-in solutions are context-bound.
- `ProviderContainer` overrides make testing trivial тАФ swap any provider with a fake in one line, no widget tree needed
- Code-gen (`@riverpod`) reduces boilerplate ~70% and enforces patterns at compile time

**Tradeoff:** Steeper learning curve than setState/ValueNotifier. Justified by async streaming complexity and multi-platform scope.

### 4. Material 3 as Base + shadcn_flutter for Surface Components
**Decision:** Use Material 3 (built-in, zero deps) for structural primitives (Scaffold, NavigationRail, routing scaffolding, layout) and `shadcn_flutter` for all visible surface components (Button, Card, Dialog, Input, Badge).
**Rationale:**
- Material 3 is idiomatic Flutter for routing, scaffold, and layout тАФ fighting it adds friction with no visual payoff
- shadcn_flutter is designed for simultaneous integration with Material; they don't conflict
- Surface components (buttons, cards, dialogs) are where visual consistency with the web shadcn aesthetic matters тАФ shadcn_flutter handles this
- Geist typography, slate colour scheme, Lucide icons, dark/light mode all come from shadcn_flutter
- This matches the web: Next.js (structural) + shadcn/ui (surface components)

**Tradeoff:** Adds ~10MB to bundle; worth it for design coherence with the web app.

### 5. Feature-First Folder Structure (Not Layer-First)
**Decision:** Organize by feature (chat/, blog/, auth/) not by layer (data/, domain/, presentation/).
**Rationale:**
- Easier to find related code (add feature = one folder)
- Self-contained; reduces cross-feature imports
- Scales better for multiple teams

**Tradeoff:** Some shared code lives in `shared/` (inevitable). Minimal impact.

### 6. Desktop as First-Class (window_manager + tray_manager)
**Decision:** Desktop UX (macOS/Windows/Linux) is priority, not afterthought.
**Rationale:**
- otherdev serves technical audience; desktop is preferred
- Single-window app doesn't need complex window management
- Keyboard shortcuts + menu bar critical for UX

**Tradeoff:** More dependencies; worth it for native feel.

### 7. Hive for Local Storage (Not SQLite by Default)
**Decision:** Use Hive for simple key-value persistence; Drift only if schema complexity grows.
**Rationale:**
- Hive is simpler for chat messages + preferences
- No SQL schema migration burden
- Drift is available if needed later

**Tradeoff:** If future features need complex queries, migrate to Drift. Simple now.

## 13. Visual Design & Theming Rules

### ThemeExtension for Design Tokens
Custom shadcn-equivalent tokens (slate palette, radius multiplier, Geist font sizes) must live in a `ThemeExtension`, not scattered across widgets. This keeps the design system centralized and dark/light-mode-safe:

```dart
@immutable
class AppTokens extends ThemeExtension<AppTokens> {
  const AppTokens({required this.cardGlow, required this.noiseOpacity});
  final Color cardGlow;
  final double noiseOpacity;

  @override
  AppTokens copyWith({Color? cardGlow, double? noiseOpacity}) =>
      AppTokens(cardGlow: cardGlow ?? this.cardGlow,
                noiseOpacity: noiseOpacity ?? this.noiseOpacity);

  @override
  AppTokens lerp(ThemeExtension<AppTokens>? other, double t) {
    if (other is! AppTokens) return this;
    return AppTokens(cardGlow: Color.lerp(cardGlow, other.cardGlow, t)!,
                     noiseOpacity: lerpDouble(noiseOpacity, other.noiseOpacity, t)!);
  }
}
// Access: Theme.of(context).extension<AppTokens>()!.cardGlow
```

### Visual Design Requirements (from rules.md)
- **Background**: subtle noise texture overlay on main background тАФ premium tactile feel
- **Cards**: multi-layered drop shadows тАФ soft, deep, "lifted" appearance
- **Interactive elements** (buttons, sliders, checkboxes): shadow with elegant color glow effect
- **Typography**: clear hierarchy тАФ hero text, section headlines, list headlines, keywords in paragraphs
- **Contrast**: WCAG 2.1 minimum 4.5:1 for normal text, 3:1 for large text (18pt+ or 14pt bold)

### Accessibility
- `Semantics` widget for all interactive elements without inherent labels
- Test with TalkBack (Android) and VoiceOver (iOS) before each release
- Dynamic text scaling тАФ UI must remain usable at 2├Ч system font size

## 14. Code Quality Rules (Flutter-Specific)

### Private Widget Classes (not helper methods)
Extract `build()` sub-trees into private `StatelessWidget` classes тАФ never into methods returning `Widget`:

```dart
// Wrong тАФ helper method
Widget _buildHeader() => Text('Header');

// Right тАФ private widget class
class _Header extends StatelessWidget {
  @override
  Widget build(BuildContext context) => Text('Header');
}
```

### `compute()` for Expensive Operations
JSON parsing of large API responses, RAG document processing, and any CPU-heavy operation must run via `compute()` to avoid blocking the UI thread:

```dart
final messages = await compute(parseMessages, jsonString);
```

### `const` Constructors
Use `const` on every widget and value where possible. Flutter's tree diffing skips `const` subtrees entirely тАФ critical for chat list performance with many message bubbles.

## 15. Testing Strategy

### Assertions
Use `package:checks` for expressive, readable assertions over raw `expect()`:
```dart
// Instead of: expect(result, equals(42));
check(result).equals(42);
check(list).isNotEmpty();
check(user.name).startsWith('A');
```

### Mock Strategy
Prefer **fakes and stubs** over mocks. Fakes implement the real interface with simplified in-memory logic тАФ they don't break when the API changes signature. Use `mocktail` only when a fake is impractical (e.g., platform channels).

```dart
// Prefer this тАФ fake implementation
class FakeChatRepository implements ChatRepository {
  final _messages = <ChatMessage>[];
  @override
  Stream<String> sendMessage(String prompt) async* { yield 'fake response'; }
}

// Over this тАФ brittle mock
final mock = MockChatRepository();
when(() => mock.sendMessage(any())).thenAnswer((_) => Stream.value('response'));
```

## 15. Risk Mitigation

### Risk: SSE Connection Drops on Mobile
**Mitigation:**
- Implement connection retry with exponential backoff
- Show reconnection UI feedback
- Save partial messages locally
- Resume stream on reconnect

### Risk: Memory Leak on Long Chat Sessions
**Mitigation:**
- Profile with DevTools memory profiler
- Implement stream cleanup (cancel subscriptions)
- Limit message history in memory (paginate old messages)
- Add memory test that runs chat for 1hr

### Risk: Desktop UI Not Responsive at Large Scales
**Mitigation:**
- Test on ultrawide (3440px) and 4K monitors
- Use MediaQuery breakpoints
- Add golden tests at desktop resolutions

### Risk: Firebase Auth Silent Failure on Some Devices
**Mitigation:**
- Test on real devices (iOS 12, iOS latest, Android 24, Android latest)
- Log all auth state changes
- Add error reporting (Sentry)

### Risk: CI/CD Complexity for 5 Platforms
**Mitigation:**
- Use Codemagic (managed CI) instead of self-hosted GitHub Actions
- Start with 1 platform (iOS), add others incrementally
- Document signing cert management in CONTRIBUTING.md
